# Architecture Review (2026-04-01)

## 概要
このプロジェクトは React Native + Expo の実装例として、`components / features / navigation / services / lib` を分離した構成を採用しています。

## 良い点

1. **責務分離の方向性が明確**
   - `app/` 配下で `components`, `features`, `navigation`, `services`, `lib` を分けており、レイヤー構成が把握しやすいです。
2. **Feature のコロケーション**
   - `app/features/...` 配下で画面・型・コンポーネントを近接配置しており、変更時の探索コストを下げています。
3. **Lint ルールでアーキテクチャを強制**
   - `features` の import 制約と `_` プレフィックスファイルの import 制約があり、設計の崩れを防ぐ仕組みがあります。
4. **Provider の集中管理**
   - App ルートで provider を集約しているため、横断関心（SafeArea, Gesture, Toast）の見通しが良いです。
5. **UI コンポーネントと Storybook 整備**
   - `stories/` があり、コンポーネントの再利用・検証基盤が用意されています。

## 課題（改善余地）

1. **`_service.ts` などの空ファイルが多い**
   - 将来拡張の意図は理解できますが、現時点では読み手にノイズを与え、実装の所在を曖昧にします。
2. **import パスが相対参照中心**
   - 深い階層で `../../../../../` が増えており、保守性が低下します。
3. **画面コンポーネントの責務肥大化の兆候**
   - `Auth` 画面のように、フォーム状態管理・API呼び出し・表示制御が同一ファイルに集中しています。
4. **ドメイン境界がまだ薄い**
   - service 層はあるものの、ユースケース単位の Application 層や domain model 層の分離は薄く、大規模化時に変更影響が拡散しやすいです。
5. **ナビゲーション定義の集約不足**
   - 画面追加時の変更点が複数ファイルに分散しやすく、規模拡大時に統制が難しくなります。

## 規模別の実用性評価

### 小規模（〜5人、〜20画面）
- **実用性: 高い**
- 現状の構成で十分に運用可能。開発速度とわかりやすさのバランスが良いです。

### 中規模（5〜15人、20〜80画面）
- **実用性: 条件付きで高い**
- 以下を先に実施すれば安定運用しやすいです。
  - import alias 導入
  - Feature ごとの public API（`index.ts`）徹底
  - 画面ロジックを hooks / usecase へ分離
  - 空ファイル削減と生成テンプレート化

### 大規模（15人以上、80画面以上）
- **実用性: 現状のままでは中〜低**
- 推奨強化:
  - feature 単位のモジュール境界（依存方向の厳格化）
  - ドメイン・アプリケーション層の明確化
  - 状態管理戦略（server state / client state）標準化
  - ADR（Architecture Decision Record）運用
  - CIで境界違反検出（import graph / dependency rules）

## 優先度付き改善案

1. **最優先**: import alias（`@/components`, `@/features` など）を導入。
2. **高**: 画面ロジックを `useXxx` hooks + service/usecase に分離。
3. **高**: 空の `_service.ts` / `_type.ts` は削除または自動生成ルールに統一。
4. **中**: navigation の route 定義を集中管理し、型を1箇所起点にする。
5. **中**: feature ごとの `README` か `module.md` を置き、責務・依存ルールを明文化。

## 総評
- サンプル/教育用途としては非常に読みやすく、実務の小〜中規模にも展開しやすい下地があります。
- 一方で、**「境界の強制」と「ロジック分離」** を先回りで整備しないと、中規模後半から急に保守負荷が上がるタイプの構成です。
